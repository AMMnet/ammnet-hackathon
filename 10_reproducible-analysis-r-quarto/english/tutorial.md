# Tutorial: Reproducible Biomedical Data Analysis Pipeline with R & Quarto

Welcome to the AMMnet Hackathon reproducible analysis tutorial! This step-by-step guide walks you through the entire clinical and epidemiological research pipeline, from installing raw requirements to generating a publication-ready manuscript draft in Microsoft Word (.docx) format aligned with PLOS ONE guidelines.

This pipeline operates under strict Standard Operating Procedures (SOPs) for workspace management, path resolution, coding conventions, statistical standards, and output layouts.

---

## 1. Prerequisites & Software Installation

Before beginning, install the following software packages on your system:

1. **R (version $\ge$ 4.0):** The core statistical computing environment. Download from the [Comprehensive R Archive Network (CRAN)](https://cran.r-project.org/).
2. **RStudio Desktop:** The recommended Integrated Development Environment (IDE). Download from [Posit](https://posit.co/download/rstudio-desktop/).
3. **Quarto CLI:** The publishing system used to compile dynamic reports. Download from [Quarto](https://quarto.org/docs/get-started/).
4. **Microsoft Word or any compatible software (e.g., LibreOffice Writer, OpenOffice Writer, Google Docs):** Required to inspect and review the final generated manuscript (.docx).

Open R or RStudio and execute the following command to install all package dependencies used in this pipeline:

## 3. Data Cleaning & Anomalies Check (`scripts/s01_clean_data.r`)

With your folder structures established, place your provided raw files (`wsdd_data.csv`, `wsdd_data_dictionary.html`, `study_protocol.pdf`, and `sap.pdf`) into their corresponding folders:

- Raw data in `data/raw/dat/`
- Raw dictionary in `data/raw/dic/`
- Protocol and SAP in `docs/`

### Implementing the Cleaning Script

Create `scripts/s01_clean_data.r`. The script executes initial data logical integrity checks:

1. **Load Raw Data:** Use `readr::read_csv()` and declare values that should be treated as missing data explicitly:

   ```r
   raw_data <- readr::read_csv(
     here::here("data", "raw", "dat", "wsdd_data.csv"),
     na = c("", "NA", ".", "N/A")
   )
   ```

2. **Examine Structure:** Check columns and shapes using `utils::glimpse(raw_data)`.
3. **Anomalies Validation:** Verify date alignments and logical bounds:
   - Check if `time_to_first_episode` exceeds `follow_up_time`.
   - Identify post-censoring events (e.g., participants lost to follow-up, but recording a diarrhoea episode date later than their study censoring date).
   - In our case, this identified exactly **10 records** containing post-censoring events.
4. **Log Errors & Export:** Log these 10 records as an RDS file to `reports/output/cleaning/cleaning_errors.rds` for the report, exclude them from the cohort, and export the remaining 1,190 clean records to `data/processed/dat/clean_data.rds`.

### Compiling the Cleaning Report

Create `reports/r01_data_cleaning.qmd` mapping to the CSL file `reports/aux/plos.csl` (the citation style file used by you target journal, in this case PLOS ONE, you can download it from <https://www.zotero.org/styles> or <https://github.com/citation-style-language/styles>). Write code blocks to load the raw cohort, processed cohort, and error logs, then display the cleaning exclusions dynamically in tables.

Run the compilation command in your terminal:

```bash
quarto render reports/r01_data_cleaning.qmd
```

This generates a clean HTML report `reports/r01_data_cleaning.html` summarizing the data cleaning lifecycle.

## 4. Cohort Processing & Variable Coding (`scripts/s02_process_data.r`)

Create `scripts/s02_process_data.r` to build the analysis-ready analytical dataset:

1. **Format Categorical Variables:** Re-code factors using `{forcats}` and establish baseline reference groups:
   - `water_source`: reference level `"Tap water"`
   - `diarrhoea_status`: reference level `"No"`
   - `sanitation`: reference level `"Improved"`
   - `socioeconomic_status`: reference level `"High"`
   - `education_head`: reference level `"Higher"`
2. **Compute Derived Variables:**
   - Censor-adjusted follow-up time: `survival_time = ifelse(is.na(time_to_first_episode), follow_up_time, time_to_first_episode)`
   - Binary event status: `event = ifelse(diarrhoea_status == "Yes", 1, 0)`
   - Net person-time at risk in years: `person_time_years = (follow_up_time - (num_episodes * 3 / 365)) / 365.25` (assuming 3 days duration per episode).
3. **Export Cohort and Dictionary:**
   - Save the processed analytical dataset to `data/processed/dat/analysis_cohort.rds`.
   - Generate a markdown-based data dictionary describing the processed fields and export it to `data/processed/dic/analysis_cohort_dictionary.md`.

## 5. Statistical Data Analysis (`scripts/s03_analise_data.r`)

Create `scripts/s03_analise_data.r` to execute the statistical model regressions outlined in the SAP:

1. **Multivariable Logistic Regression (Binary Risk):**

   ```r
   log_adj <- glm(
     diarrhoea_status ~ water_source + age + sex + sanitation + socioeconomic_status + education_head,
     data = cohort,
     family = binomial(link = "logit")
   )
   ```

2. **Multivariable Negative Binomial Regression (Recurrent Rates):**

   ```r
   nb_adj <- MASS::glm.nb(
     num_episodes ~ water_source + age + sex + sanitation + socioeconomic_status + education_head + offset(log(person_time_years)),
     data = cohort
   )
   ```

3. **Multivariable Cox Proportional Hazards (Time-to-First-Episode):**

   ```r
   cox_adj <- survival::coxph(
     survival::Surv(survival_time, event) ~ water_source + age + sex + sanitation + socioeconomic_status + education_head,
     data = cohort_survival
   )
   ```

4. **Diagnostic Checks:**
   - Assess multicollinearity using Variance Inflation Factors: `car::vif()` or manually via model matrices to ensure VIF values are within safe boundaries (< 5.0).
   - Evaluate proportional hazard assumptions using Schoenfeld residuals: `survival::cox.zph(cox_adj)`.
5. **Export Model Objects:** Save models and diagnostic results as R objects to `reports/output/analysis/` (e.g., `logistic_models.rds`, `nb_models.rds`, `cox_models.rds`, and `vif_results.rds`).

## 6. Developing the Manuscript Drafts (`reports/r03_manuscript.qmd` & `reports/r03_manuscript_no_crossref.qmd`)

To provide both a fully cross-referenced document and a warning-free Word draft, we create two versions of the manuscript:

1. **`reports/r03_manuscript.qmd` (Cross-Referenced HTML & Word):** Configured to output both formats. In HTML, Quarto's native cross-referencing (`#| label: tbl-...` and `#| tbl-cap: ...` inside code chunks, cited with `@tbl-...`) is fully supported, resulting in an interactive web manuscript. In Word, it compiles but may trigger XML cell mapping recovery warnings on opening.
2. **`reports/r03_manuscript_no_crossref.qmd` (Warning-Free Word):** Configured exclusively for Word. It uses manual headings and textual referencing to avoid Word's XML parser warnings.

### Key Manuscript Construction Guidelines

To build these documents and understand the differences, review these formatting conventions:

#### A. Setup Chunk Placement

Place the R `setup` code chunk at the **top** of the document, immediately following the YAML metadata header and **before the `# Abstract` section**:

````yaml
---
title: "Water Source and Diarrhoeal Disease: A Prospective Cohort Study"
author:
  - name: "Arsénio Nhacolo"
    affiliations:
      - name: "Centro de Investigação em Saúde de Manhiça"
    corresponding: true
format:
  docx:
    toc: false
    number-sections: false
bibliography: aux/references.bib
csl: aux/plos.csl
---

```{r}
#| label: setup
#| include: false
# (Load libraries, import cohort/model .rds files, and compute inline variables here)
```

# Abstract

**Background:** ...
**Results:** Over follow-up, `r diarrhoea_cases` cases was observed...

````
This enables inline variables (like `` `r diarrhoea_cases` ``) to be defined *before* they are compiled in the Abstract text.

#### B. Resolving the Word XML "Ambiguous cell mapping" Warning
When compiling to Word, the interaction between Quarto's table cross-referencing markers (`#| label: tbl-...` and `#| tbl-cap: ...`) and the `flextable` table objects creates malformed OpenXML tags (missing paragraph tags before `</tc>`), causing Word to display a recovery warning.

*To resolve this:*

1. **Remove** `#| label: tbl-...` and `#| tbl-cap: ...` from table chunks.
2. Use non-cross-reference chunk labels (e.g., `#| label: demographics-table`).
3. Place manual Markdown headers directly above the chunks:
   ````markdown
   **Table 1: Baseline characteristics of the study population.**
   ```{r}
   #| label: demographics-table
   #| echo: false

   cohort |>
     dplyr::select(...) |>
     gtsummary::tbl_summary(...) |>
     gtsummary::add_overall(last = FALSE) |>
     gtsummary::bold_labels() |>
     gtsummary::as_flex_table()
   ```

   ````
4. Cite tables in the text using standard text (e.g. `"summarized in Table 1."` instead of `"summarized in @tbl-demographics."`).

#### C. Survival Plot Integration in Word
When rendering to Word, the risk table alignment inside `survminer::ggsurvplot` often fails. Set `risk.table = FALSE` and render only the main ggplot component (`print(p_km$plot)`) to output a clean, high-resolution vector image:
````markdown
````{r}
#| label: fig-survival
#| fig-cap: "Kaplan-Meier survival curves showing diarrhoea-free probability over follow-up days."
#| fig-width: 8
#| fig-height: 6
#| fig-dpi: 300
#| echo: false

fit_km <- survival::survfit(
  survival::Surv(survival_time, event) ~ water_source,
  data = cohort_survival
)

p_km <- survminer::ggsurvplot(
  fit_km,
  data = cohort_survival,
  pval = TRUE,
  conf.int = TRUE,
  palette = c("#2c3e50", "#e74c3c"),
  risk.table = FALSE,
  ggtheme = ggplot2::theme_minimal(),
  legend.labs = c("Tap water", "Well water"),
  legend.title = "Water Source",
  xlab = "Follow-up Time (Days)",
  ylab = "Diarrhoea-Free Probability"
)

print(p_km$plot)
````

````

## 7. Pipeline Compilation & Verification

### Running the Steps Individually

To run and verify the entire pipeline, run the R scripts and render the Quarto documents in order:

```bash
# 1. Clean raw dataset and log anomalies
Rscript scripts/s01_clean_data.r

# 2. Factor-code baseline references and build analytical cohort
Rscript scripts/s02_process_data.r

# 3. Model regression coefficients and test diagnostics
Rscript scripts/s03_analise_data.r

# 4. Render the data cleaning report (HTML)
quarto render reports/r01_data_cleaning.qmd

# 5. Render the detailed epidemiological analysis report (HTML)
quarto render reports/r02_data_analysis.qmd

# 6. Render the manuscripts
# Compile the cross-referenced version (HTML & Word):
quarto render reports/r03_manuscript.qmd

# Compile the warning-free Word version:
quarto render reports/r03_manuscript_no_crossref.qmd
```

If you prefer using the RStudio graphical interface rather than the command line:

- **For R Scripts (`scripts/s01_clean_data.r`, etc.):** Open the script file in RStudio and click the **Run** button at the top right of the editor pane (or press `Ctrl+Enter` / `Cmd+Return` to run selected lines). You can also run the entire script from the R console using `source("scripts/s01_clean_data.r")`.

## 8. Code Quality, Formatting, and Linting

To maintain a professional, readable, and consistent codebase, the project enforces strict styling guidelines using the R packages `{styler}` and `{lintr}`.

### Standard Coding Guidelines Enforced

- **Assignment:** Always use `<-` for assignment (use `=` only within functions).
- **Piping:** Use the native R pipe operator `|>` exclusively.
- **Operators:** Surround all infix operators (`<-`, `|>`, `==`, `+`) with spaces.
- **Line Length:** Maintain a maximum line length of 80 characters.
- **Package Conflicts:** Prefix unique functions with their respective package name (e.g., `dplyr::select()` vs. `MASS::select()`) to avoid namespace conflicts.

### Running Formatting & Linting Manually

You can run these tools directly inside R or RStudio:

```r
# Automatically format R scripts under the 'scripts' directory
styler::style_dir("scripts")

# Run style audits and syntax check warnings
lintr::lint_dir("scripts")
```

### Running via the Root Utility Script

Alternatively, you can run the code quality check directly from the command line using the provided [`run_style_and_lint.r`](run_style_and_lint.r) script:

```bash
Rscript run_style_and_lint.r
```

This utility will style and check your code files and output a clean status summary.

### Handling Linter Exceptions

When writing multi-line string literals (e.g., raw Markdown table data dictionaries in R scripts), standard line wrapping would break the output formatting. In these cases, you can disable the linter warnings locally using `# nolint` comments:

```r
# nolint start: line_length_linter
dictionary_content <- "
| Variable | Description |
| :--- | :--- |
| `water_source` | Primary drinking water source |
"
# nolint end: line_length_linter
```
